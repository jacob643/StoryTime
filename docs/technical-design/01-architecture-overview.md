# Architecture Overview

## System Context

Story Time is an adventure typing game driven by a Large Language Model (LLM). The player types paragraphs generated by an LLM, and their typing speed determines how the story progresses — faster typing leads to more positive outcomes, slower typing leads to setbacks.

```
+-------------------+         HTTP/JSON          +-------------------+
```

## Key Design Principles

1. **Local-first** — The primary LLM backend is local (Ollama), with optional custom endpoints for advanced users. No external service required.

2. **Provider abstraction** — All LLM interactions go through a common interface so new backends (OpenAI, Anthropic, LM Studio) can be added without changing game logic.

3. **Speed-driven narrative** — Typing speed per paragraph maps to an outcome tier (very negative → very positive). The story prompt is selected based on this tier, creating a natural feedback loop.

4. **Stateless frontend** — The browser is purely a presentation layer. All game state, story history, and configuration live on the backend.

5. **Incremental distribution** — Starts as a local dev server, graduates to a PyInstaller executable, with potential for Tauri wrapping later.

## Data Flow — One Game Turn

```
Player types paragraph
        |
        v
Browser sends input + metadata to POST /api/chat
        |
        v
Backend calculates speed (CPM/WPM), accuracy
        |
        v
Speed mapped to outcome tier (0-4):
  Very Negative | Negative | Neutral | Positive | Very Positive
        |
        v
Prompt template selected based on tier +
  story history appended as context
        |
        v
Request sent to LLM provider (Ollama/custom)
        |
        v
LLM returns next paragraph
        |
        v
Paragraph validated (length, safety)
        |
        v

        |
        v

        |
        v
[repeat]
```

## Core Abstractions

### Game Session
- Holds story history (list of paragraphs with metadata)
- Tracks current paragraph index
- Maintains player settings for the session

### Outcome Tier
- Computed from typing speed relative to player baseline or absolute threshold
- 5 tiers matching the 5 prompt templates already defined in the codebase

### Prompt Template
- A parameterized prompt string that drives the next story paragraph
- Receives: story history, outcome tier description, character/setting context
- Output: a single paragraph continuation

### LLM Provider
- Interface with methods: `list_models()`, `generate()`, `is_available()`
- Implementations: OllamaProvider, CustomOpenAIProvider
- Future: AnthropicProvider, LmStudioProvider

## Technology Choices

| Component | Choice | Rationale |
|---|---|---|
| Backend framework | FastAPI (Python) | Async-native, excellent for LLM I/O, auto-docs, lightweight |
| Frontend | Vanilla HTML/CSS/JS | Zero build step, easy to serve statically, can evolve to SPA later |
| LLM provider | Ollama (primary) + OpenAI-compatible (secondary) | Ollama is the most accessible local LLM runner; OpenAI-compatible covers most other backends |
| Packaging | PyInstaller | Single executable for distribution, supports Windows/Mac/Linux |
| Async HTTP | httpx | Modern async HTTP client for Python, needed for non-blocking LLM calls |

## Ports & Networking

| Service | Default Port | Notes |
|---|---|---|
| FastAPI backend | 8000 | Serves UI + API, configurable via flag or env var |
| Ollama | 11434 | Standard Ollama port, configured in backend settings |
| Custom endpoint | user-defined | Any OpenAI-compatible API |
